# REGLAS GLOBALES DE DESARROLLO - PROYECTO: TIENDALOCAL

## 1. Contexto General del Sistema

- **Objetivo:** Sistema de Gestión de Ventas y Delivery Local para un Kiosco.
- **Arquitectura:** Mantenible, escalable y sostenible. Separación clara de responsabilidades: Frontend (React + Vite), Backend (FastAPI), Base de Datos (Supabase/PostgreSQL).
- **Idioma:** Todo el código fuente debe incluir comentarios exhaustivos y detallados en ESPAÑOL. La documentación técnica generada también debe ser en español.

## 2. Stack Tecnológico y Restricciones de Código

- **Backend:** Python con FastAPI. Estructura limpia (routers, schemas con Pydantic, services).
- **Base de Datos:** Supabase (PostgreSQL).
  - **REGLA CRÍTICA DE DB:** Maximizar el uso del motor de base de datos. Los cálculos de stock, validaciones de saldo deudor y flujos complejos de negocio se deben resolver mediante **Procedimientos Almacenados (PL/pgSQL)**, **Triggers** e **Índices** optimizados. El backend (FastAPI) debe actuar principalmente como un orquestador/API Gateway seguro.
  - **Bajas Lógicas:** Queda prohibido el uso de la sentencia `DELETE` física en tablas principales. Todo borrado debe ser una actualización del atributo `estado` a 'Inactivo' o 'Cancelado'.
- **Frontend:** React + Vite. Interfaces completamente **responsivas** (Mobile-first o adaptables al 100% para pantallas móviles de repartidores/cajeros y monitores de escritorio). Usar una biblioteca de estilos moderna (ej. Tailwind CSS).

## 3. Control de Versiones (Git)

- **Repositorio:** https://github.com/JoseMPB13/tiendaLocal.git
- **Flujo:** Al finalizar el desarrollo completo y exitoso de cada submódulo o característica funcional, la IA debe estructurar y sugerir un comando de commit claro y descriptivo (siguiendo Conventional Commits, ej: `feat(ventas): add stored procedure for checkout and inventory update`), listo para que el usuario confirme y suba al repositorio.

## 4. Documentación y Calidad

- Cada archivo de código debe comenzar con un encabezado descriptivo.
- Las funciones complejas deben documentarse explicando: Qué hace, Parámetros de entrada, Valores de retorno y Excepciones/Errores posibles.
- Generar y actualizar archivos Markdown (`.md`) para documentar la arquitectura del sistema a medida que se avanza.
