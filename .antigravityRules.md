MANUAL DE REGLAS CONSOLIDADAS DE DESARROLLO Y OPERACIÓN - PROYECTO: TIENDALOCAL
1. Contexto General del Sistema
Objetivo del Sistema: Sistema de Gestión de Ventas y Delivery Local para un Kiosco (Tienda Margarita).

Arquitectura Base: Separación clara y estricta de responsabilidades en tres capas: Frontend (React + Vite), Backend (FastAPI), Base de Datos (Supabase/PostgreSQL).

Idioma Obligatorio: Todo el código fuente debe incluir comentarios exhaustivos y detallados en ESPAÑOL. Toda la documentación técnica generada en archivos Markdown (.md) también debe redactarse estrictamente en español.

2. Stack Tecnológico y Restricciones de Código
A) Capa de Base de Datos (Supabase / PostgreSQL)
Regla Crítica "DB-First": Maximizar el uso del motor de base de datos. Los cálculos de stock, validaciones de saldo deudor, límites de crédito, flujos complejos de negocio y auditorías se resuelven mediante Procedimientos Almacenados (PL/pgSQL), Triggers e Índices optimizados. El backend actúa principalmente como un orquestador o API Gateway seguro.

Prohibición de Borrado Físico (Bajas Lógicas): Queda estrictamente prohibido el uso de la sentencia DELETE física en las tablas principales del sistema. Todo borrado debe ser una baja lógica mediante la actualización del atributo estado a 'Inactivo' o 'Cancelado', disparando de forma atómica las funciones trigger de reversión de stock y saldos en el motor SQL.

B) Capa de Backend (Python / FastAPI)
Estructura Limpia: Organización modular estricta dividida en enrutadores (routers), esquemas de validación (schemas con Pydantic) y servicios de orquestación (services).

Mapeo de Errores Robustos: Capturar las excepciones nativas de la base de datos utilizando la clase postgrest.exceptions.APIError y filtrar mediante códigos de error SQLSTATE de Postgres (ej. P0001, P0002), eliminando comparaciones frágiles basadas en cadenas de texto plano.

Control Estricto de Trailing Slashes (Barras Finales): Para blindar los endpoints de tipo POST y PUT contra errores 404 Not Found provocados por la barra inclinada final que envía el cliente web, los endpoints deben declararse duplicando los decoradores de ruta de ser necesario:

Python
@router.post("")
@router.post("/")
Sanitización y Serialización JSON: Antes de enviar diccionarios de datos al cliente de Supabase (.insert() o .update()), se deben convertir explícitamente todos los objetos de tipo UUID nativos de Python a cadenas de texto (str) para evitar mueras de ejecución del tipo TypeError: Object of type UUID is not JSON serializable.

C) Capa de Frontend (React + Vite)
Responsividad Absoluta: Las interfaces deben ser 100% responsivas y adaptables, priorizando un enfoque Mobile-first para las pantallas de repartidores y cajeros, sin descuidar monitores de escritorio.

Consistencia Estética y Estilos: Usar única y exclusivamente la biblioteca moderna Tailwind CSS. Queda prohibido mezclar estilos inline extensos o usar dimensiones fijas (como height: 100vh) que rompan la experiencia en dispositivos móviles o táctiles. Las tablas deben transformarse dinámicamente en tarjetas (cards) en pantallas pequeñas.

Reactividad de Mapas (Leaflet): La inicialización e interacción con Leaflet debe controlarse de forma imperativa mediante ganchos (useEffect y useRef) para asegurar total estabilidad con React 19. El re-centrado de la cámara (map.setView) y el reposicionamiento del marcador deben reaccionar en caliente ante cambios externos del formulario (como el evento onBlur de un enlace).

Extractor de Coordenadas Universal: El analizador de enlaces de geolocalización debe ser agnóstico al dominio. Debe buscar patrones numéricos decimales (Lat/Lng) mediante expresiones regulares tolerantes en cualquier parte de la URL, permitiendo procesar dominios de prueba o desarrollo (googleusercontent.com, placeholders) y activando la geocodificación inversa de Nominatim de inmediato.

3. Metodología Obligatoria de Persistencia y Edición de Archivos
Modificación Física Real: El agente del IDE tiene la obligación estricta de aplicar las modificaciones de código directamente sobre los archivos físicos del espacio de trabajo. No debe limitarse a entregar resúmenes de cambios, explicaciones conceptuales o guías en texto plano.

Gestión Dual de Scripts SQL: Cuando el desarrollo requiera realizar modificaciones o adiciones en la estructura o programabilidad de la base de datos, el agente debe proceder bajo dos fases de persistencia:

Script Incremental: Generar un script SQL independiente en la carpeta de borradores (ej: scratch/migracion_*.sql) con los comandos ALTER, CREATE OR REPLACE o restricciones oportunas para aplicarse de inmediato en la base de datos activa sin pérdida de información.

Actualización de Maestros: Modificar e incorporar de manera limpia esas mismas sentencias al final de los archivos maestros de inicialización del repositorio (schema.sql y programmability.sql), asegurando que el sistema pueda desplegarse desde cero en cualquier momento.

4. Control de Versiones (Git) e Automatización
Automatización de Ejecución: Una vez que el desarrollo de una característica o corrección se complete con éxito, y se verifique mediante compilación en caliente (py_compile en backend / npm run build en frontend) que el sistema no tiene errores de sintaxis, el agente debe ejecutar de forma automática el comando de confirmación en la terminal del workspace.

Estándar del Mensaje: Los comandos de confirmación deben estructurarse bajo la convención internacional de Conventional Commits, pero redactando el mensaje descriptivo ESTRICTAMENTE EN ESPAÑOL:

Ejemplo de corrección: fix(clientes): corregir reactividad del mapa y auto-rellenado de direccion en evento onblur

Ejemplo de nueva funcionalidad: feat(productos): integrar reabastecimiento e historial de compras en la vista de productos

5. Documentación y Calidad de Código
Encabezados Descriptivos: Cada archivo de código modificado o creado debe comenzar con un bloque de comentario superior que sirva de encabezado, explicando de forma clara su propósito, dependencias y módulo al que pertenece.

Documentación de Funciones: Las funciones o procedimientos complejos deben incluir un bloque explicativo detallando: Qué hace la función, Parámetros de entrada, Valores de retorno y Excepciones/Errores controlados.

Actualización de Manuales Markdown: Tras culminar exitosamente los cambios, el agente debe actualizar los archivos Markdown (.md) correspondientes dentro de la carpeta docs/ (tales como esquema_base_datos.md, endpoints_api.md o arquitectura_frontend.md), dejando registro técnico de las nuevas entidades, endpoints o componentes integrados.
